# Samba AD DC vs FreeIPA: ключевые различия

## 🎯 Основное назначение

| Аспект | **Samba AD DC** | **FreeIPA** |
|--------|----------------|-------------|
| **Цель** | Эмуляция контроллера домена Microsoft Active Directory | Централизованное управление идентичностью для Linux/Unix-сред |
| **Основной протокол** | SMB/CIFS, LDAP, Kerberos (совместимый с AD) | LDAP, Kerberos, DNS, с акцентом на Unix-атрибуты |
| **Совместимость с Windows** | ✅ Полная: клиенты Windows можно присоединить к домену, работают групповые политики | ❌ Ограниченная: Windows-клиенты нельзя присоединить напрямую к домену FreeIPA [[1]] |

---

## 🔧 Архитектурные различия

### Samba AD DC
- Работает на уровне функциональности леса **Windows Server 2008 R2** (с версии 4.19 — экспериментальная поддержка более новых уровней) [[2]]
- Использует встроенный LDAP-бэкенд и KDC (Heimdal или экспериментально MIT Kerberos)
- Поддерживает **репликацию между контроллерами домена** как в настоящем AD
- Включает полноценную поддержку **Group Policy Objects (GPO)** для управления Windows-клиентами
- Интегрированный DNS-сервер (SAMBA_INTERNAL или BIND9_DLZ)

### FreeIPA
- Построен на базе **389 Directory Server (LDAP)**, **MIT Kerberos**, **BIND** (DNS) и **Dogtag PKI**
- Предоставляет веб-интерфейс и мощные CLI-инструменты для управления
- Поддерживает **HBAC (Host-Based Access Control)** и **Sudo-правила** вместо GPO
- Имеет встроенную поддержку **двухфакторной аутентификации** и интеграции с OTP
- Поддерживает **Kerberos proxy (MS-KKDCP)** для безопасного развёртывания в DMZ [[1]]

## 🌐 Протоколы и интеграция

| Функция | Samba AD DC | FreeIPA |
|---------|-------------|---------|
| **Присоединение Windows-клиентов** | ✅ Да, через стандартный механизм AD | ❌ Нет [[1]] |
| **Присоединение Linux-клиентов** | ✅ Через `sssd` + `realmd` | ✅ Нативно, через `ipa-client-install` |
| **Групповые политики (GPO)** | ✅ Полная поддержка | ❌ Нет (есть альтернативы: HBAC, sudo, Ansible) |
| **RFC2307 (Unix-атрибуты в AD)** | ✅ Опционально (`--use-rfc2307`) | ✅ Встроено по умолчанию |
| **Доверительные отношения (Trusts)** | ✅ С другими лесами AD | ⚠️ Только с доменами AD через SSSD/Winbind, не нативно |
| **PKI / Сертификаты** | ⚠️ Через внешние решения | ✅ Встроенный Dogtag CA |
| **Web UI** | ❌ Нет (только сторонние инструменты) | ✅ Полноценный веб-интерфейс |

## 🛠️ Сценарии использования

### Выбирайте **Samba AD DC**, если:
- Вам нужна **замена Microsoft AD** в гетерогенной среде (Windows + Linux)
- Требуется поддержка **присоединения Windows-рабочих станций** и применения групповых политик
- Вы хотите использовать стандартные инструменты управления AD (RSAT, PowerShell)
- Важен опыт миграции с существующего AD на open-source решение

> ⚠️ **Важно**: Не рекомендуется использовать Samba DC одновременно как файловый сервер в продакшене с несколькими контроллерами домена [[2]].

### Выбирайте **FreeIPA**, если:
- Ваша инфраструктура преимущественно **Linux/Unix-ориентирована**
- Нужен централизованный менеджмент пользователей, хостов, SSH-ключей, sudo-правил
- Требуется встроенная **поддержка сертификатов (PKI)** и двухфакторной аутентификации
- Важен удобный **веб-интерфейс** для администрирования
- Вы планируете интеграцию с облачными провайдерами идентичности (через плагины)

## 💡 Практический совет

> 🔄 **Можно ли использовать вместе?**  
> Да! Некоторые организации разворачивают **FreeIPA как основную систему управления идентичностью для Linux**, а **Samba AD DC — для поддержки унаследованных Windows-систем**, настраивая одностороннее доверие или синхронизацию через инструменты вроде `ipa-adtrust-install`. Однако это увеличивает сложность архитектуры и требует глубокого понимания обоих решений [[1]].

Если вы работаете в IT-среде и выбираете между этими решениями, начните с анализа:
1. Какой % клиентов — Windows?
2. Нужны ли вам групповые политики?
3. Есть ли требования к встроенному PKI?
4. Насколько важна простота администрирования через веб-интерфейс?

Ответы на эти вопросы помогут сделать обоснованный выбор. 🎯
