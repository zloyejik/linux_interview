Восстановление удалённого файла с помощью `lsof` **возможно только в одном случае**: файл был удалён (`rm`), но **остался открытым хотя бы одним процессом**. В этом случае данные всё ещё находятся в памяти и доступны через виртуальную файловую систему `/proc`.

> ⚠️ Если все процессы закрыли файл, его дескрипторы из `/proc` исчезнут, и этот метод не сработает. В таком случае потребуются специализированные утилиты (`extundelete`, `photorec`, `testdisk`) или резервные копии.

---

### 📌 Пошаговая инструкция

#### 1. Найдите удалённый, но открытый файл
```bash
sudo lsof +L1
```
`+L1` показывает файлы, у которых количество ссылок (link count) меньше 1, то есть удалённые из директории, но всё ещё открытые процессами.

В выводе обратите внимание на столбцы:
```
COMMAND   PID   USER   FD   TYPE DEVICE SIZE/OFF   NODE NAME
nginx    1234   root   4w   REG   8,1     10240  567890 /var/log/nginx/access.log (deleted)
```
- `PID` = `1234`
- `FD` = `4` (число перед буквой режима: `4r`, `4w`, `4u` → берём только цифру)
- `(deleted)` в конце подтверждает удаление

#### 2. Убедитесь, что файл действительно доступен
```bash
ls -l /proc/1234/fd/4
```
Вы увидите символическую ссылку вида:
```
lrwx------ 1 root root 64 Apr 13 10:00 /proc/1234/fd/4 -> '/var/log/nginx/access.log (deleted)'
```
Это и есть прямой доступ к данным файла.

#### 3. Восстановите файл
```bash
sudo cp /proc/1234/fd/4 /tmp/recovered_access.log
```
или
```bash
sudo cat /proc/1234/fd/4 > /tmp/recovered_access.log
```

#### 4. Проверьте результат
```bash
ls -lh /tmp/recovered_access.log
head /tmp/recovered_access.log
```
Содержимое должно совпадать с ожидаемым. Обратите внимание: новый файл получит **новый inode**, а старые права/владелец могут отличаться (их можно вернуть через `chown`/`chmod`).

---

### 🔍 Где здесь inode?
В выводе `lsof` столбец `NODE` содержит номер inode удалённого файла. Он нужен не для восстановления через `/proc`, а для:
- проверки через `stat /proc/PID/fd/FD`
- поиска в метаданных ФС (например, `debugfs` для ext4)
- сопоставления с резервными копиями

Сам процесс восстановления через `lsof` опирается на **PID + FD**, а не напрямую на inode.

---

### ⚠️ Критические ограничения и рекомендации
1. **Не перезагружайте систему и не перезапускайте процесс** до восстановления. При закрытии дескриптора данные помечаются как свободные.
2. Если процесс активно пишет в файл, восстановленная копия будет моментальным "снимком". Для консистентных данных лучше временно остановить процесс или использовать `lsof` на моменте, когда запись приостановлена.
3. Файлы на `tmpfs`, `procfs`, `sysfs` не подлежат такому восстановлению (они не хранятся на диске).
4. После восстановления проверьте целостность: `md5sum`, `file`, или специфичные для формата утилиты.

---

### 🆘 Что делать, если файл уже закрыт всеми процессами?
| ФС       | Утилиты восстановления                          |
|----------|------------------------------------------------|
| ext4/ext3| `extundelete`, `e2fsck -b`, `debugfs`           |
| XFS      | `xfs_undelete`, `photorec`                      |
| btrfs    | `btrfs restore`, `snapper` (если есть снимки)   |
| Любая    | `testdisk` + `photorec` (сигнатурный анализ)    |

💡 **Совет на будущее:** используйте `tmux`/`screen` для сессий, настройте снапшоты LVM/btrfs/zfs, и храните резервные копии. Удаление через `rm` в Linux необратимо без открытых дескрипторов или специализированных инструментов.
