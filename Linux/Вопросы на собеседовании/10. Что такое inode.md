**Inode** (index node) — это фундаментальная структура данных в файловых системах Unix-подобных операционных систем (ext4, XFS, Btrfs и др.), которая хранит метаданные о файле или директории.

### 🔹 Что хранится в inode:

| Поле | Описание |
|------|----------|
| **Inode number** | Уникальный идентификатор структуры в рамках файловой системы |
| **Тип файла** | Обычный файл, директория, символическая ссылка, устройство и т.д. |
| **Права доступа** | Чтение/запись/выполнение для владельца, группы и остальных (rwx) |
| **Владелец и группа** | UID и GID пользователя, которому принадлежит файл |
| **Размер файла** | В байтах |
| **Временные метки** | `atime` (последний доступ), `mtime` (изменение содержимого), `ctime` (изменение метаданных) |
| **Ссылки** | Количество жёстких ссылок на этот inode |
| **Блоки данных** | Указатели на физические блоки диска, где хранится содержимое файла |

### 🔹 Что НЕ хранится в inode:
- **Имя файла** — оно хранится в директории, которая связывает имя с номером inode
- **Путь к файлу** — это логическая конструкция, собираемая из структуры директорий

### 🔹 Практические примеры:

```bash
# Посмотреть информацию об inode файла
ls -i filename          # показать номер inode
stat filename           # подробная информация об inode

# Найти файлы по номеру inode
find / -inum 123456

# Проверить использование inode на диске
df -i                   # показывает занятое/свободное количество inode
```

### 🔹 Важные особенности:

1. **Жёсткие ссылки** (`ln file1 file2`) — разные имена, указывающие на один и тот же inode. Удаление одного имени не удаляет данные, пока счётчик ссылок > 0.

2. **Символические ссылки** — это отдельные файлы со своим inode, содержащие путь к другому файлу.

3. **Лимит inode** — при форматировании диска создаётся фиксированное количество inode. Даже если есть свободное место, но закончились inode, новые файлы создать нельзя.

4. **Производительность** — поиск по inode быстрее, чем по имени, так как не требует обхода дерева директорий.

### 🔹 Почему это важно для IT-специалиста:
- Понимание inode помогает диагностировать проблемы с «местом на диске» (когда `df` показывает место, а файлы не создаются — возможно, исчерпаны inode)
- Объясняет поведение жёстких ссылок и восстановление файлов
- Критично для работы с бэкапами, контейнерами и низкоуровневыми операциями в Linux




В файловой системе **ext4** (default для большинства дистрибутивов Linux) inode реализован с рядом важных оптимизаций по сравнению с ext2/ext3. Ниже разбор архитектуры, хранения данных и особенностей работы.

---

### 📐 1. Организация inode в ext4
| Параметр | Значение |
|----------|----------|
| **Размер inode** | 256 байт по умолчанию (в ext2/ext3 было 128) |
| **Расположение** | Таблица inode (`inode table`) находится в начале каждой Block Group |
| **Количество** | Рассчитывается при `mkfs.ext4` (обычно ~1 inode на 16 КБ пространства). Изменить позже нельзя без переформатирования |
| **Нумерация** | Глобально уникальна в рамках ФС. Inode `2` — корневая директория `/` |

ext4 делит диск на **Block Groups** (обычно по 128 МБ). В каждой группе хранится:
- суперблок (резервный)
- дескрипторы групп
- таблица inode
- битовые карты блоков и inode
- данные файлов

---

### 🗃️ 2. Структура inode (256 байт)
Первые 128 байт совместимы с ext2/ext3, остальные 128 байт зарезервированы под новые фичи ext4.

| Поле | Назначение |
|------|------------|
| `i_mode` | Тип файла + права (rwx, setuid, sticky и т.д.) |
| `i_uid` / `i_gid` | Владелец и группа |
| `i_size` / `i_size_high` | Размер файла (поддержка до 16 ТБ в стандартном режиме, до 1 ЭБ с `large_file`) |
| `i_atime`, `i_mtime`, `i_ctime`, `i_crtime` | Временные метки (включая время создания) |
| `i_links_count` | Счётчик жёстких ссылок |
| `i_blocks` | Количество занятых 512-байтных секторов (не обязательно совпадает с `i_size`) |
| `i_flags` | Флаги ФС (`immutable`, `append`, `nodump`, `extent` и др.) |
| `i_block[15]` | **Ключевое поле**: до 60 байт под указатели на данные или extents |
| `i_extra_isize` | Сколько дополнительных байт (сверх 128) действительно используется |
| `i_checksum` | Контрольная сумма inode (фича `metadata_csum`) |

---

### 🌳 3. Как хранятся данные файла: Extents вместо косвенных указателей
В ext2/ext3 использовались прямые + одинарные/двойные/тройные косвенные указатели. В ext4 по умолчанию включена фича **`extent`**.

**Extent** — это запись вида:  
`[начальный_логический_блок, длина, начальный_физический_блок]`

✅ Преимущества:
- Один extent заменяет сотни косвенных указателей
- Меньше фрагментации, выше скорость последовательного чтения/записи
- Меньше накладных расходов на метаданные

📦 Хранение в inode:
- В `i_block[15]` помещается до **4 extents** (по 15 байт каждый + заголовок)
- Если файл больше или сильно фрагментирован, строится **дерево extents** (extent tree), корень которого лежит в `i_block`, а ветви — в отдельных блоках диска

🔍 Проверить extents файла:
```bash
filefrag -v /path/to/file
# или
debugfs -R "stat <123456>" /dev/sda1 | grep -i extent
```

---

### ⚡ 4. Оптимизации ext4 для маленьких файлов
| Фича | Описание |
|------|----------|
| **Inline Data** (`inline_data`) | Если файл ≤ ~60 байт, его содержимое хранится прямо в `i_block` внутри inode. Экономит блок диска и ускоряет чтение |
| **Fast Symlinks** | Короткие симлинки хранят путь прямо в inode, не занимая отдельный блок |
| **Tail Packing** | Неиспользованное место в последнем блоке данных может быть отведено под метаданные или xattr |

> Inline data включается при создании ФС: `mkfs.ext4 -O inline_data /dev/sdX` или включается динамически для отдельных файлов через `chattr +I` (ядро ≥ 5.13).

---

### 🛡️ 5. Журналирование и целостность
- ext4 по умолчанию журналирует **только метаданные** (`data=ordered`)
- Изменения inode (права, размер, время, указатели) сначала пишутся в журнал, затем на диск
- После crash журнал replay'ится, inode возвращается в консистентное состояние
- Режим `data=journal` (журналирует и данные) включается вручную, но снижает производительность

---

### 🏷️ 6. Расширенные атрибуты (xattr)
ACL, capabilities, SELinux/AppArmor контексты, user.xattr хранятся:
1. **Внутри inode**, если хватает места в `i_extra_isize`
2. **Во внешних блоках** с хеш-таблицей (фича `extra_isize` + `xattr`)
3. С контрольной суммой (фича `metadata_csum` предотвращает silent corruption)

---

### 🚀 7. Производительность и служебные фичи
| Фича | Назначение |
|------|------------|
| **Flex Block Groups** (`flex_bg`) | Группы объединяются в пулы, метаданные распределяются равномернее, уменьшается hot-spot |
| **Lazy Inode Table Initialization** | При `mkfs` таблица не затирается нулями мгновенно, а очищается фоновым потоком `ext4lazyinit` |
| **Inode Readahead** | Ядро предзагружает соседние inode при сканировании директорий (ускоряет `ls -R`, `find`, бэкапы) |
| **Delayed Allocation** | Блоки выделяются не при `write()`, а при flush, что позволяет ядру оптимальнее размещать extents |

---

### 🛠️ 8. Диагностика и низкоуровневый доступ
```bash
# Общая информация о ФС (размер inode, количество, stride, фичи)
dumpe2fs -h /dev/sda1 | grep -i inode

# Детальный дамп конкретного inode
debugfs -R "stat <131073>" /dev/sda1

# Проверка использования inode
df -i /

# Включение/выключение фич (требует unmount)
tune2fs -O ^extent /dev/sda1   # отключить extents (не рекомендуется)
tune2fs -l /dev/sda1           # список активных фич
```

---

### ⚠️ Важные нюансы для администратора
1. **Лимит inode фиксирован**. Если `df -i` показывает 100%, новые файлы создать нельзя, даже если есть место. Лечится только удалением файлов или переформатированием с `-N` или `-i`.
2. **Удаление файла** = уменьшение `i_links_count` до 0 → помечение inode как свободного → очистка `i_block`/extents. Данные физически остаются до перезаписи.
3. **Жёсткие ссылки** делят один inode. Разные имена → один `i_links_count` > 1.
4. **`i_blocks` vs `i_size`**: `i_blocks` считает выделенные 512-байтные сектора (включая разреженные участки и метаданные), `i_size` — логический размер файла.
