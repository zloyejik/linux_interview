# Как работает Kerberos

**Kerberos** — это сетевой протокол аутентификации, разработанный в MIT, который позволяет пользователям и службам безопасно подтверждать свою личность в незащищённой сети с помощью системы билетов (tickets) и доверенной третьей стороны.

---

## 🔑 Основные компоненты

| Компонент                           | Описание                                                |
| ----------------------------------- | ------------------------------------------------------- |
| **Client**                          | Пользователь или служба, запрашивающая доступ к ресурсу |
| **Server**                          | Ресурс, к которому клиент хочет получить доступ         |
| **KDC (Key Distribution Center)**   | Доверенный центр распределения ключей, состоящий из:    |
| ├─ **AS (Authentication Server)**   | Проверяет личность пользователя при входе               |
| └─ **TGS (Ticket Granting Server)** | Выдаёт билеты для доступа к конкретным службам          |

---

## 🔄 Процесс аутентификации (упрощённо)

### Шаг 1: Запрос TGT (Ticket Granting Ticket)
```
Client → AS: "Я пользователь alice, хочу войти в систему"
AS → Client: 
  - TGT (зашифрован секретным ключом TGS)
  - Сессионный ключ (зашифрован паролем alice)
```
> Клиент расшифровывает сессионный ключ, введя пароль. **Пароль никогда не передаётся по сети.**

### Шаг 2: Запрос сервисного билета
```
Client → TGS: 
  - TGT (полученный ранее)
  - Authenticator (время + данные, зашифрованные сессионным ключом)
  - Имя запрашиваемой службы (например, fileserver)

TGS → Client:
  - Сервисный билет (зашифрован ключом службы)
  - Новый сессионный ключ для общения с сервисом
```

### Шаг 3: Доступ к сервису
```
Client → Server:
  - Сервисный билет
  - Новый Authenticator (зашифрован сессионным ключом)

Server → Client:
  - Подтверждение (опционально, для взаимной аутентификации)
```
> Сервер расшифровывает билет своим ключом, проверяет Authenticator и предоставляет доступ.

---

## 🔐 Ключевые особенности безопасности

✅ **Пароли не передаются по сети** — только хеши/ключи  
✅ **Билеты имеют ограниченное время жизни** (по умолчанию ~8–10 часов)  
✅ **Mutual authentication** — и клиент, и сервер могут подтвердить подлинность друг друга  
✅ **Защита от replay-атак** благодаря временным меткам и одноразовым аутентификаторам  
✅ **Централизованное управление доступом** через KDC

---

## ⚠️ Ограничения и уязвимости

- **Единая точка отказа**: если KDC недоступен, аутентификация невозможна
- **Требует точной синхронизации времени** (разница обычно ≤5 минут)
- **Уязвим к атакам на слабые пароли** (offline brute-force на зашифрованный TGT)
- **Golden/Silver Ticket attacks** — если злоумышленник получит ключ KDC или службы

---

## 💡 Где используется?

- 🔹 **Active Directory** (Windows-домены)
- 🔹 **Linux/Unix-системы** (через PAM, krb5)
- 🔹 **Hadoop, PostgreSQL, SSH, NFS** и другие корпоративные сервисы
- 🔹 **Single Sign-On (SSO)** в гетерогенных средах

---

## 🧪 Пример команды в Linux

```bash
# Получить TGT
kinit alice@REALM.COM

# Просмотреть полученные билеты
klist

# Получить доступ к сервису (автоматически)
ssh fileserver.company.com

# Удалить билеты
kdestroy
```
