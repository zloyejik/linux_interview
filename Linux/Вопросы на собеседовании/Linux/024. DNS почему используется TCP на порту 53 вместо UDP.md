# DNS: почему используется TCP на порту 53 вместо UDP?

На самом деле, **DNS использует и UDP, и TCP на порту 53** — выбор протокола зависит от задачи. 🔄

## 🚀 Почему по умолчанию используется UDP?

Большинство DNS-запросов идут через **UDP**, потому что это:

| Преимущество | Объяснение |
|-------------|-----------|
| **Низкая задержка** | Нет рукопожатия (3-way handshake) — запрос и ответ за 1 круговой путь (RTT) [[2]] |
| **Меньше нагрузки на сервер** | Нет состояния соединения — сервер не хранит информацию о клиентах [[1]] |
| **Эффективность** | Типичный ответ DNS (~150 байт) легко помещается в один UDP-пакет [[21]] |

## 🛡️ Когда DNS переключается на TCP?

DNS использует **TCP на порту 53** в следующих случаях:

### 1. **Ответ не помещается в UDP-пакет**
- Оригинальный лимит UDP по RFC 1035: **512 байт**
- При превышении сервер устанавливает флаг **TC (Truncated)**, и клиент повторяет запрос через TCP [[12]]
- Частые причины большого ответа:
  - **DNSSEC** — криптографические подписи значительно увеличивают размер [[6]]
  - **Много записей** — популярные домены с десятками A/AAAA-записей
  - **Крупные TXT-записи** — SPF, DKIM, верификация доменов
  - **Цепочки CNAME** — несколько уровней перенаправления

### 2. **Передача зон (Zone Transfer)**
- Операции **AXFR/IXFR** (копирование всей зоны между серверами) **всегда используют TCP** [[4]]
- Причины: надёжность, порядок доставки, большой объём данных

### 3. **Требования безопасности или политики сети**
- Некоторые межсетевые экраны или прокси требуют TCP для инспекции трафика [[12]]

## 🔁 Как работает переключение UDP → TCP?

```
1. Клиент отправляет запрос через UDP (порт 53)
2. Сервер видит, что ответ > лимита → ставит флаг TC
3. Клиент получает усечённый ответ → открывает TCP-соединение
4. Повторяет запрос через TCP → получает полный ответ
```

Современные резолверы кэшируют информацию о том, какие домены требуют TCP, чтобы избежать лишних попыток через UDP [[2]].

## ⚠️ Что если TCP на порту 53 заблокирован?

Если межсетевой экран блокирует TCP/53:
- Большие ответы (DNSSEC, IPv6) не доставляются
- Клиенты видят **таймауты** или **очень медленное разрешение имён** [[6]]
- Может происходить фрагментация IP-пакетов, которую некоторые сети отбрасывают

✅ **Рекомендация**: разрешите **и UDP, и TCP на порту 53** в правилах фаервола для надёжной работы DNS [[21]].

## 🔐 EDNS и современные расширения

Расширение **EDNS(0)** позволяет согласовать больший размер UDP-пакета (до 4096 байт), что снижает необходимость переключения на TCP. Однако:
- Не все сетевые устройства корректно обрабатывают большие UDP-пакеты
- Рекомендуется использовать консервативный размер буфера (например, 1232 байта) для избежания фрагментации [[12]]

---

> 💡 **Итог**: UDP — для скорости и массовых запросов, TCP — для надёжности при больших ответах и служебных операциях. Оба протокола на порту 53 необходимы для полноценной работы современной инфраструктуры DNS.
