Процесс в состоянии **D (Uninterruptible Sleep)** — это «особый случай» в Linux: он ждёт завершения критической операции (обычно ввод-вывод: диск, сеть, NFS, драйвер) и **игнорирует все сигналы**, включая `kill -9`. Убить его напрямую нельзя — нужно устранять причину блокировки.

Вот пошаговый алгоритм действий:

---

## 🔍 Шаг 1: Диагностика — что держит процесс?

```bash
# 1. Узнать PID и команду
ps -eo pid,stat,comm | grep '^.* D'

# 2. Посмотреть, какие файлы/устройства открыты
lsof -p <PID>
# или
ls -l /proc/<PID>/fd

# 3. Проверить стек вызовов ядра (покажет, где именно ждёт)
cat /proc/<PID>/stack
```
Пример вывода `stack`:
```
[<0>] __wait_on_bit+0x...      # ждёт завершения I/O
[<0>] blk_wait_io+0x...        # блок-устройство
[<0>] nfs_file_read+0x...      # NFS-операция
```
📌 Ключевые слова: `nfs`, `blk`, `scsi`, `ext4`, `btrfs`, `fuse`, `virtio` — укажут на подсистему.

```bash
# 4. Проверить системные логи на ошибки устройства
dmesg -T | tail -30
journalctl -k -n 50 --no-pager  # ошибки ядра

# 5. Проверить нагрузку на диск
iostat -x 1 3
iotop -oPa                      # покажет, кто реально ждёт I/O
```

---

## 🛠 Шаг 2: Решения в зависимости от причины

### 🟡 Причина 1: Медленный/зависший диск или файловая система
**Симптомы**: `stack` показывает `blk_wait_io`, `ext4_sync_file`, `btrfs_wait_ordered`; `dmesg` содержит `I/O error`, `sector`, `sdX: failed`.

**Что делать**:
```bash
# Проверить SMART диска
smartctl -a /dev/sdX

# Проверить файловую систему (только на размонтированном!)
umount /dev/sdX
fsck /dev/sdX

# Если диск "отвалился" в RAID — пересобрать массив
cat /proc/mdstat
mdadm --manage /dev/md0 --add /dev/sdX
```
⚠️ **Не монтируйте принудительно (`mount -o remount,ro`)** без бэкапа — риск потери данных.

---

### 🟡 Причина 2: Зависший NFS / сетевая ФС
**Симптомы**: в `stack` — `nfs_wait_request`, `rpc_wait_for_completion`; процесс висит при доступе к `/mnt/nfs`.

**Что делать**:
```bash
# Проверить доступность сервера
ping <nfs_server>
rpcinfo -p <nfs_server>

# Попробовать "мягко" размонтировать (если не занято)
umount -l /mnt/nfs    # lazy unmount
# или принудительно (рискованно!)
umount -f /mnt/nfs

# Если не помогает — перезапустить nfs-клиент
systemctl restart rpcbind
```
💡 На будущее: монтируйте NFS с опцией `soft` (вместо `default hard`), чтобы процессы не висели вечно:
```
server:/share /mnt/nfs nfs soft,timeo=30,retrans=2 0 0
```

---

### 🟡 Причина 3: Проблемный драйвер / устройство (USB, GPU, виртуализация)
**Симптомы**: `stack` указывает на драйвер (`usb_storage`, `virtio_blk`, `nvidia`); процесс появился после подключения устройства.

**Что делать**:
```bash
# Перезагрузить модуль ядра (осторожно!)
lsmod | grep <driver>
rmmod <driver>   # может не сработать, если занят
modprobe <driver>

# Для USB: пересканировать шину
echo 1 > /sys/bus/usb/devices/usbX/authorized  # отключить
echo 0 > /sys/bus/usb/devices/usbX/authorized  # включить
```
⚠️ Если драйвер не выгружается — может потребоваться перезагрузка.

---

### 🟡 Причина 4: Долгая, но легальная операция (бэкап, дефрагментация, rsync)
**Симптомы**: процесс в `D`, но `iotop` показывает активный I/O, `dmesg` чист, диск работает.

**Что делать**:
- ✅ **Подождать** — если операция важная и прогресс есть.
- 📊 Мониторить прогресс:
  ```bash
  watch -n 5 'grep -E "R|D" /proc/<PID>/status'
  iotop -p <PID>
  ```

---

## ☠️ Если ничего не помогает: крайние меры

| Мера | Когда применять | Риски |
|------|----------------|-------|
| **Перезагрузка системы** | Процесс критичен, диск/сеть не восстанавливаются | Потеря несохранённых данных, downtime |
| **Magic SysRq (без перезагрузки)** | Ядро поддерживает `CONFIG_MAGIC_SYSRQ` | Может "отпустить" зависшие I/O: <br> `echo w > /proc/sysrq-trigger` (показать задачи) <br> `echo t > /proc/sysrq-trigger` (stack trace всех) |
| **kdump / crash** | Для пост-мортем анализа в production | Требует предварительной настройки |

🔧 Включите Magic SysRq (если отключён):
```bash
echo 1 > /proc/sys/kernel/sysrq
# Затем безопасно "разбудить" задачи:
echo w > /proc/sysrq-trigger   # покажет блокированные процессы
```

## 🛡 Профилактика: как избежать зависаний в `D`

1. **Для дисков**:
   - Используйте `hdparm -tT` / `smartctl` для мониторинга здоровья.
   - Настройте `vm.dirty_ratio`, `vm.dirty_background_ratio`, чтобы избежать долгих `sync`.
   - Для SSD: включите TRIM, следите за wear-leveling.

2. **Для NFS**:
   - Монтируйте с `soft`, `timeo`, `retrans`.
   - Используйте `autofs` для автоматического (де)монтирования.

3. **Мониторинг**:
   ```bash
   # Алерт при появлении процессов в D > 30 сек
   while true; do
     ps -eo pid,stat,comm | awk '$2 ~ /^D/ {print strftime("%F %T"), $0}' >> /var/log/d_state.log
     sleep 10
   done
   ```

4. **Ядро и драйверы**:
   - Обновляйте ядро и firmware устройств — многие баги `D`-состояний исправляются в новых версиях.

---

> ⚠️ **Важно**: Процесс в `D` — это **не баг**, а защитный механизм ядра. Принудительное завершение могло бы повредить данные. Поэтому ядро не позволяет его убить — пока операция не завершится *корректно*.

Если хотите — пришлите вывод `ps`, `stack` и `dmesg` для конкретного процесса, помогу разобрать причину точнее. 🛠️
