# Linux Load Average: что это и как считается

## 📊 Что такое Load Average

**Load Average** (средняя нагрузка) — это метрика, показывающая среднее количество процессов, которые находятся в состоянии **выполнения** или **ожидания** выполнения в системе за определённый период времени [[1]].

```bash
$ uptime
 16:48:24 up 4:11, 1 user, load average: 0.75, 0.50, 0.30
                              │        │     │     │
                              │        │     │     └─ 15-минутное среднее
                              │        │     └─────── 5-минутное среднее  
                              │        └───────────── 1-минутное среднее
                              └────────────────────── три числа нагрузки
```

## 🔢 Что именно учитывается

В Linux load average включает процессы в двух состояниях:

| Состояние | Описание | Обозначение |
|-----------|----------|-------------|
| **TASK_RUNNING** | Процесс выполняется или готов к выполнению на CPU | `R` |
| **TASK_UNINTERRUPTIBLE** | Процесс ожидает ресурс (обычно диск/I/O), не реагирует на сигналы | `D` |

> ⚠️ **Важно**: В отличие от других ОС, где load average показывает только нагрузку на CPU, в Linux это **"системная нагрузка"**, включающая также ожидание дискового I/O и некоторых блокировок [[1]].

## 🧮 Как считается: формула

Load average вычисляется как **экспоненциально затухающее скользящее среднее** (EWMA — Exponentially Weighted Moving Average) [[8]][[13]].

### Базовая формула:
```
load(t) = load(t-1) × e^(-5/60) + nr_active × (1 - e^(-5/60))
```

Где:
- `t` — текущий момент времени
- `load(t-1)` — предыдущее значение нагрузки
- `nr_active` = `nr_running + nr_uninterruptible` (активные процессы)
- `e^(-5/60)` ≈ 0.92 — коэффициент затухания для 1-минутного интервала
- Вычисления происходят **каждые 5 секунд** (`LOAD_FREQ`)

### Константы в ядре Linux:
```c
// include/linux/sched/loadavg.h
#define EXP_1   1884    // 1/exp(5sec/1min) как фиксированная точка
#define EXP_5   2014    // 1/exp(5sec/5min)  
#define EXP_15  2037    // 1/exp(5sec/15min)
```

## 📈 Интерпретация значений

### Относительно количества ядер:
```
Load / CPU_cores < 0.7  → Система недогружена, есть запас
Load / CPU_cores ≈ 1.0  → Оптимальная загрузка
Load / CPU_cores > 1.0  → Есть очередь задач, возможна задержка
Load / CPU_cores > 5.0  → Серьёзные проблемы с производительностью
```

### Динамика трёх чисел:
| Паттерн | Значение |
|---------|----------|
| `1min > 5min > 15min` | Нагрузка **растёт** 📈 |
| `1min < 5min < 15min` | Нагрузка **падает** 📉 |
| Все значения близки | Стабильная нагрузка ➡️ |

## 🔍 Практические примеры

### Пример 1: Нормальная работа
```bash
$ cat /proc/loadavg
0.45 0.52 0.48 2/345 12345
# │    │    │    │  │    └─ последний PID
# │    │    │    │  └───── всего процессов
# │    │    │    └──────── запущенные/всего
# │    │    └───────────── 15-мин среднее
# │    └────────────────── 5-мин среднее  
# └─────────────────────── 1-мин среднее
```

### Пример 2: Высокая нагрузка из-за диска
```bash
# Процесс в состоянии "D" (ожидание диска) тоже увеличивает load:
$ ps aux | awk '$8 ~ /D/ {print}'
root  1234  0.0  0.1  12345  6789 ?  D  10:30   0:05 [jbd2/sda1-8]

# Даже если CPU свободен, load будет высоким:
$ uptime
 load average: 8.50, 7.20, 6.80  # ← из-за ожидания диска!
$ mpstat 1
# CPU    %idle
# all    95.2   # ← CPU почти свободен!
```

## 🛠️ Где посмотреть и как использовать

### Просмотр:
```bash
uptime                    # Кратко
cat /proc/loadavg         # Детально + дополнительная информация
top                       # В реальном времени
htop                      # Удобный интерфейс
```

### Мониторинг в скриптах:
```bash
#!/bin/bash
LOAD_1=$(cut -d' ' -f1 /proc/loadavg)
CPU_CORES=$(nproc)
LOAD_PER_CORE=$(echo "$LOAD_1 / $CPU_CORES" | bc -l)

if (( $(echo "$LOAD_PER_CORE > 1.0" | bc -l) )); then
    echo "⚠️  Высокая нагрузка: $LOAD_PER_CORE задач на ядро"
fi
```

## ⚠️ Частые заблуждения

| Миф | Реальность |
|-----|------------|
| "Load = загрузка CPU в %" | Это **количество задач**, а не процент. Может быть >1.0 [[8]] |
| "1мин = среднее за 60 секунд" | Это **экспоненциальное** среднее с весом последних данных [[13]] |
| "Высокий load = проблема" | Не всегда: если это короткий всплеск или только на одном ядре — может быть нормой |
| "Load учитывает только CPU" | В Linux учитываются также процессы, ожидающие диск и некоторые блокировки [[1]] |

## 🔧 Более точные метрики для диагностики

Когда load high, используйте дополнительные инструменты [[1]]:

```bash
# CPU-нагрузка по ядрам:
mpstat -P ALL 1

# Процессы, потребляющие CPU:
pidstat 1

# Очередь планировщика (saturation):
vmstat 1          # колонка 'r'
perf sched record # детальный анализ

# Дисковая активность:
iostat -x 1

# Процессы в состоянии D:
ps -eo state,pid,cmd | grep "^D"
```

## 📚 История и контекст

Интересный факт: включение `TASK_UNINTERRUPTIBLE` в расчёт load average произошло в **Linux 0.99.14 (ноябрь 1993)** по предложению Маттиаса Урлихса. Его аргумент: *"Если заменить быстрый диск свопинга на медленный, система станет медленнее, но load average уменьшится — это нелогично"* [[1]].

> 💡 **Вывод**: Load average — полезная метрика для **трендов и относительных сравнений**, но для точной диагностики всегда дополняйте её другими инструментами мониторинга.
