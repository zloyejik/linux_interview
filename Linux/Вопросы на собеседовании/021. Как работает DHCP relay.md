# Как работает DHCP Relay Agent

**DHCP Relay Agent** (агент ретрансляции DHCP) — это компонент сетевого оборудования (обычно маршрутизатора или L3-коммутатора), который позволяет передавать сообщения DHCP между клиентами и серверами, находящимися в разных подсетях [[1]].

---

## 🔍 Зачем нужен DHCP Relay?

Проблема: клиенты DHCP отправляют запросы через **широковещательные пакеты** (broadcast) с адресом назначения `255.255.255.255`. Маршрутизаторы по умолчанию **не пересылают широковещательный трафик** между интерфейсами, поэтому клиент в одной подсети не может «достучаться» до DHCP-сервера в другой [[2]].

Решение: включение функции DHCP Relay на маршрутизаторе позволяет преобразовать широковещательный запрос клиента в **одноадресный (unicast)** пакет и переслать его серверу [[4]].

---

## 🔄 Пошаговый процесс работы

### 1️⃣ DHCP Discover (клиент → сервер)
```
Клиент отправляет:  [Broadcast] DHCP Discover
                    ↓
Relay Agent получает пакет на интерфейсе подсети клиента
                    ↓
Relay Agent модифицирует пакет:
  • Destination IP: 255.255.255.255 → IP DHCP-сервера
  • Source IP: 0.0.0.0 → IP uplink-интерфейса Relay Agent
  • giaddr (Gateway IP): 0.0.0.0 → IP интерфейса, с которого получен запрос
  • Destination MAC: FF:FF:FF:FF:FF:FF → MAC DHCP-сервера
                    ↓
Relay Agent отправляет: [Unicast] DHCP Discover → DHCP-сервер
```

### 2️⃣ DHCP Offer (сервер → клиент)
```
DHCP-сервер получает запрос, видит giaddr и определяет подсеть клиента
                    ↓
Сервер выбирает свободный IP из соответствующего пула
                    ↓
Сервер отправляет: [Unicast] DHCP Offer → giaddr (Relay Agent)
                    ↓
Relay Agent проверяет флаг Broadcast:
  • Если 0: заменяет dest IP на выданный клиенту адрес (unicast)
  • Если 1: заменяет dest IP на 255.255.255.255 (broadcast)
                    ↓
Relay Agent отправляет: [Local Broadcast/Unicast] DHCP Offer → клиент
```

### 3️⃣ DHCP Request и DHCP Ack
Аналогичный процесс: клиент вещает Request → Relay преобразует в unicast → сервер отвечает → Relay доставляет ответ клиенту [[2]].

---

## 📋 Ключевые поля в пакете

| Поле | Исходное значение (клиент) | После Relay Agent |
|------|---------------------------|------------------|
| **Destination IP** | `255.255.255.255` | IP DHCP-сервера |
| **Source IP** | `0.0.0.0` | IP uplink-интерфейса Relay |
| **giaddr** | `0.0.0.0` | IP интерфейса Relay, принявшего запрос |
| **Destination MAC** | `FF:FF:FF:FF:FF:FF` | MAC DHCP-сервера |

> ⚠️ Поле **giaddr (Gateway IP Address)** критически важно: именно по нему сервер определяет, из какой подсети пришёл запрос, и выбирает соответствующий пул адресов [[7]].

---

## 🔄 Обновление и освобождение адреса

Важное уточнение: 
- **Обновление аренды (Renewal)** и **освобождение адреса (Release)** происходят через **unicast-сообщения** напрямую от клиента к серверу.
- В этих случаях DHCP Relay **не участвует**, так как пакеты не являются широковещательными и маршрутизируются обычным образом [[7]].

## ⚙️ Пример конфигурации

### На Linux (dhcrelay):
```bash
dhcrelay -i eth0 -i eth1 192.168.100.10
# eth0 — интерфейс к клиентам, eth1 — к серверу, 192.168.100.10 — адрес DHCP-сервера
```
[[3]]

### На Cisco:
```cisco
interface Vlan10
  ip helper-address 192.168.100.10
```

### На Windows Server:
DHCP Relay Agent настраивается через консоль **Routing and Remote Access** [[8]].

## ✅ Преимущества использования DHCP Relay

1. **Централизация**: один сервер может обслуживать множество подсетей
2. **Экономия ресурсов**: не нужно разворачивать сервер в каждой подсети
3. **Упрощение управления**: политики и пулы адресов настраиваются в одном месте
4. **Масштабируемость**: легко добавлять новые подсети без изменения архитектуры

---

## ⚠️ Важные замечания

- На межсетевых экранах между Relay и сервером необходимо разрешить трафик на **порт 67/UDP** (DHCP) [[11]].
- Relay Agent должен иметь статический IP-адрес на интерфейсе, обращённом к серверу.
- Для отладки полезно анализировать поле **giaddr** в дампах трафика (Wireshark).
