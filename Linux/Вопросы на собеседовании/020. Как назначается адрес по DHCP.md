DHCP (Dynamic Host Configuration Protocol) — это протокол, который автоматически назначает сетевые параметры устройствам в сети. Процесс получения адреса состоит из четырёх основных этапов, известных как **DORA**:

### 🔄 Этапы процесса DORA

| Этап            | Сообщение      | Описание                                                                       |
| --------------- | -------------- | ------------------------------------------------------------------------------ |
| **D**iscover    | `DHCPDISCOVER` | Клиент рассылает широковещательный запрос в сеть: «Есть ли здесь DHCP-сервер?» |
| **O**ffer       | `DHCPOFFER`    | Сервер(ы) отвечают предложением: «Я могу выдать вам адрес X, маска Y, шлюз Z»  |
| **R**equest     | `DHCPREQUEST`  | Клиент выбирает одно из предложений и запрашивает конкретные параметры         |
| **A**cknowledge | `DHCPACK`      | Сервер подтверждает выделение адреса и отправляет окончательные настройки      |

### 📦 Что ещё передаёт DHCP кроме IP-адреса:
- Маску подсети (Subnet Mask)
- Шлюз по умолчанию (Default Gateway)
- DNS-серверы
- Аренду времени (Lease Time) — срок, на который выдаётся адрес
- Опционально: NTP-серверы, доменное имя, параметры для VoIP/PXE и др.

### ⏱️ Аренда адреса (Lease Time)
- Адрес выдаётся не навсегда, а на определённое время
- При истечении 50% срока аренды клиент пытается продлить её, отправив `DHCPREQUEST` напрямую серверу
- Если продление не удалось, при 87,5% срока клиент переходит в состояние rebinding и рассылает запрос всем серверам
- По истечении срока аренды адрес возвращается в пул

### 🔍 Дополнительные сценарии:
- **DHCPNAK** — сервер отклоняет запрос (например, если клиент переместился в другую подсеть)
- **DHCPRELEASE** — клиент добровольно освобождает адрес (при корректном завершении работы)
- **Static/DHCP Reservation** — сервер всегда выдаёт один и тот же адрес конкретному MAC-адресу

### 🛡️ Безопасность и особенности:
- В больших сетях используют **DHCP Relay Agent**, чтобы запросы из разных подсетей доходили до центрального сервера
- Для защиты от поддельных серверов применяют **DHCP Snooping** на коммутаторах
- В беспроводных сетях и IoT часто используют короткие сроки аренды для эффективного управления пулом адресов

Если вы работаете с настройкой DHCP-сервера (например, на Linux — `isc-dhcp-server` или `Kea`, на Windows Server — роль DHCP), важно правильно настроить пулы адресов, исключения, опции и политики аренды.

# На каком уровне osi работает dhcp

DHCP работает на **7-м уровне модели OSI — Прикладном (Application Layer)**.

### 📊 Почему именно 7-й уровень?

| Критерий | Объяснение |
|----------|-----------|
| **Архитектура** | DHCP построен по модели «клиент-сервер», что характерно для прикладных протоколов |
| **Транспорт** | Использует **UDP** (порты **67** — сервер, **68** — клиент) как транспортный механизм |
| **Формат данных** | Сообщения DHCP имеют собственную структуру с опциями, типами сообщений и полями — как у других прикладных протоколов |
| **Назначение** | Предоставляет конфигурационную информацию приложениям и ОС, а не занимается непосредственно маршрутизацией или кадровой инкапсуляцией |

### 🔄 Взаимодействие с другими уровнями

```
Уровень 7 (Application)  → DHCP-сообщения (DISCOVER, OFFER, REQUEST, ACK)
Уровень 4 (Transport)    → UDP, порты 67/68
Уровень 3 (Network)      → IP (на этапе передачи; изначально клиент работает без IP)
Уровень 2 (Data Link)    → Ethernet, широковещательные кадры (FF:FF:FF:FF:FF:FF)
```

### ⚠️ Частая путаница

Многие ошибочно относят DHCP к 3-му уровню, потому что:
- Он назначает **IP-адреса** (функция сетевого уровня)
- Работает **до получения клиентом IP-адреса**, используя широковещание

Однако: **протокол классифицируется по тому, как он реализован, а не по тому, какие параметры он настраивает**. Поскольку DHCP — это прикладной сервис с собственной логикой, форматом сообщений и архитектурой клиент-сервер, он относится к **Layer 7**.

### 📚 Аналогия с другими протоколами

| Протокол | Уровень OSI | Примечание |
|----------|-------------|------------|
| DHCP     | 7 (Application) | Настройка сетевых параметров |
| DNS      | 7 (Application) | Разрешение имён |
| HTTP     | 7 (Application) | Передача веб-контента |
| UDP      | 4 (Transport)   | Транспорт для DHCP |
| IP       | 3 (Network)     | Адресация и маршрутизация |

> 💡 **Запомните**: Если протокол использует UDP/TCP и имеет собственную семантику сообщений — он, скорее всего, прикладной (Layer 7), даже если управляет параметрами нижних уровней.
